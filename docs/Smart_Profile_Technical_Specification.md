## 1. プロジェクト概要

### 1.1 プロダクトビジョン

**Smart Profile**は、「学生の生活を豊かにするプロダクト」をコンセプトとした、日常の記録を自動でナレッジ管理し、自己分析を支援するアプリケーションです。本アプリケーションのコアバリューは、ユーザーが意識的に記録をつける負担を軽減し、音声認識や画像認識などのAI技術を用いて日常の活動を自動的にデータ化・蓄積することにあります。

蓄積されたデータは、就職活動における自己分析、自己成長の可視化、他者とのコミュニケーション支援、さらにはAIによるマッチングなど、多岐にわたる目的で活用されます。従来、学生が手動で行っていた「日記をつける」「自己分析シートを埋める」「ポートフォリオを作成する」といった作業を、テクノロジーの力で自動化・高度化することが本プロダクトの根幹です。

## 2. ターゲットユーザーと解決すべき課題

### 2.1 ターゲットユーザー

本アプリケーションは、人生の大きな意思決定（進路選択、就職活動、キャリア形成）を控えた学生を主要なターゲットとしています。対象年齢は6歳（小学1年生）から22〜26歳（社会人になるまでの年齢）と幅広く設定されていますが、メインユーザーは大学生です。

| ペルソナ | 特徴 | 主な課題 |
| :--- | :--- | :--- |
| **キャリア志向型** | 就職活動を控え、自己分析や経験の棚卸しに悩んでいる大学生 | 自分の経験を言語化できない、ポートフォリオが作れない |
| **成長志向型** | 学生のうちに何か成長したいが、何をすべきか分からない | やりたいことが見つからない、目標が立てられない |
| **交友関係拡大型** | 新しい友達や恋人が欲しいが、きっかけがない | 初対面でのコミュニケーションが苦手、気の合う人が見つからない |
| **モチベーション低下型** | 目標はあるが、日々の努力が可視化されずモチベーションが続かない | 進捗が分からない、タスク管理ができない |
| **保護者** | 子どもの成長記録を残したいが、記録をつける余裕がない | 記録が大変で継続できない |

### 2.2 解決すべき課題の詳細

Figmaボード上の課題分析フレームワークでは、「誰が → どんな場面で → 何に困っていて → なぜ今の方法ではダメで → それが解決されると何が変わるか」という構造で課題が整理されています。以下に、主要な課題とその解決アプローチを示します。

| # | 課題 | 現状の問題 | Smart Profileによる解決 |
| :---: | :--- | :--- | :--- |
| 1 | 記録が大変 | 日記やログをつけるのが面倒で続かない | 音声・画像認識による自動記録で記録の負担をゼロにする |
| 2 | 自己分析が困難 | 自分の経験が言語化されておらず、「自分とは何か」を説明できない | 蓄積データからAIが自動で自己分析レポートを生成する |
| 3 | 経験の埋没 | 自分の経験が自分の中に眠っていて、可視化・説明ができない | ナレッジグラフとして経験を構造化し、いつでも参照可能にする |
| 4 | 目標と進捗の不可視化 | 目標に対してどの程度努力しているか分からない | モチベーショングラフの自動生成で進捗を可視化する |
| 5 | タスク管理の形骸化 | 日常のタスク管理がおろそかになる | 会話や通知からタスクを自動抽出し、外部ツールと連携する |
| 6 | コミュニケーションの壁 | 友達や恋人が欲しいが、きっかけがない | AIマッチングとプロフィール共有で出会いを支援する |
| 7 | 生活への追われ | 一人暮らしで生活に追われ、やりたいことを見失う | 自動記録により「振り返り」の時間を確保し、意思決定を支援する |

## 3. 競合分析

Figmaボード上で言及されている競合サービスは以下の通りです。

| サービス名 | 概要 | Smart Profileとの差別化ポイント |
| :--- | :--- | :--- |
| **YouMeet** | 学生向けのプロフィール共有・マッチングサービス | Smart Profileは「自動記録」をコアとし、手動入力の負担を排除する点で差別化 |
| **ProField** | プロフィール・ポートフォリオ作成ツール | Smart ProfileはAIによる動的なプロフィール自動生成を行い、常に最新の状態を維持する |

Smart Profileの最大の差別化要因は、**ユーザーが意識的に情報を入力する必要がない**という点です。音声認識、画像認識、通知解析などのマルチモーダルAI技術を活用し、日常生活そのものをデータソースとすることで、他サービスでは実現できない「自動的に成長するプロフィール」を提供します。

## 5. 機能仕様

Figmaボードで定義された機能ツリーに基づき、各機能の詳細仕様を定義します。機能は4つの階層（コア機能、技術手段、主要機能、価値提供機能）で構成されています。

### 5.1 コア機能：自動ナレッジ管理と自己分析

本アプリケーションの中核となる機能です。ユーザーの日常を自動的に記録し、AIが分析・構造化します。

**機能ID**: F-CORE-001

**機能概要**: デバイスのマイク、カメラ、各種センサーからデータを自動収集し、音声認識・画像認識・自然言語処理を用いてナレッジとして蓄積する。蓄積されたナレッジから、ユーザーの自己分析レポートを自動生成する。

**詳細仕様**:

環境データの自動収集においては、スマートフォンのマイクやカメラを活用し、ユーザーの明示的な許可に基づいて、会話や環境音、画像をバックグラウンドで収集します。収集頻度や対象はユーザーが細かく設定でき、「授業中のみ」「特定の場所でのみ」といったコンテキストベースの制御も可能とします。

マルチモーダル認識処理では、収集した音声データは音声認識（Speech-to-Text）によりテキスト化され、画像データは画像認識によりコンテキスト（場所、物体、活動内容）が抽出されます。これらの処理はプライバシー保護のため、可能な限りエッジデバイス（スマートフォン上）で一次処理を行い、匿名化されたデータのみをクラウドに送信します。

自動タグ付けと分類においては、抽出されたデータは自然言語処理を用いて「学習」「趣味」「タスク」「人間関係」「健康」などのカテゴリに自動分類され、ナレッジグラフとしてデータベースに蓄積されます。

### 5.2 プロフィール機能

蓄積されたデータをもとに、ユーザーの多面的なプロフィールを自動構築します。

**機能ID**: F-PROF-001

| サブ機能 | 機能ID | 説明 |
| :--- | :--- | :--- |
| **動的プロフィール生成** | F-PROF-001a | 過去の経験、スキル、興味関心がAIにより自動抽出され、常に最新の状態にアップデートされるプロフィールを生成する |
| **ステータス表示** | F-PROF-001b | 現在の活動状況、気分、学習の進捗などをリアルタイムで表示する |
| **ポートフォリオ化** | F-PROF-001c | 学校の成績、課外活動、就職活動のための情報を一元管理し、外部に公開可能なポートフォリオとして出力する |
| **デジタル名刺作成** | F-PROF-001d | プロフィール情報から、初対面のコミュニケーションに役立つデジタル名刺を自動生成する |
| **プロフィール共有** | F-PROF-001e | 生成されたプロフィールを他のユーザーと共有する機能。共有範囲（全体公開/友人のみ/非公開）を設定可能 |

### 5.3 タスク・目標管理機能

日常のタスク管理とモチベーション維持を支援する機能群です。

**機能ID**: F-TASK-001

自動タスク抽出機能（F-TASK-001a）では、会話履歴やスマートフォンの通知から、実行すべきタスクをAIが自動的に抽出し、ToDoリストに追加します。たとえば、「明日までにレポートを提出しなければ」という会話が検出された場合、「レポート提出」というタスクが期限付きで自動生成されます。

モチベーショングラフ機能（F-TASK-001b）では、過去の活動量や達成度を分析し、モチベーションの推移をグラフ化して可視化します。週次・月次・年次のトレンドを確認でき、モチベーションが低下している時期にはアプリからの励ましや提案が表示されます。

外部ツール連携機能（F-TASK-001c）では、抽出されたタスクや予定を、Google Apps Script（GAS）などを経由してGoogleカレンダーやGoogle ToDoなどのWorkspaceツールと自動同期します。これにより、既存のワークフローを変更することなく、Smart Profileの恩恵を受けることができます。

### 5.4 マッチング・コミュニケーション支援機能

ユーザー間のつながりを創出し、コミュニケーションを円滑にする機能群です。

**機能ID**: F-MATCH-001

| サブ機能 | 機能ID | 説明 |
| :--- | :--- | :--- |
| **AIマッチング** | F-MATCH-001a | AIが解析したプロフィール情報（興味、価値観、活動履歴）に基づき、気の合う友人や恋人候補を自動で提案する |
| **企業マッチング** | F-MATCH-001b | 企業に対して、学生の実際の活動履歴に基づくプロフィールを提示し、適切な企業やインターンシップをレコメンドする |
| **ロールモデル提示** | F-MATCH-001c | ユーザーの目標や志向に合わせて、参考となる先輩やロールモデルのプロフィールを閲覧できる機能 |
| **初対面支援** | F-MATCH-001d | 共有されたプロフィール情報をもとに、初対面の相手との共通点や話題を提案する機能 |

### 5.5 情報一元管理機能

学生生活に関わるあらゆる情報を一箇所で管理する機能です。

**機能ID**: F-INFO-001

学校の成績、資格、課外活動の実績、就職活動の進捗など、学生生活に関わる情報を一元的に管理します。これらの情報は手動入力に加え、連携した外部サービス（大学のポータルサイト等）からの自動取得も将来的に検討します。管理された情報はプロフィール生成やポートフォリオ出力の素材として活用されます。

## 6. データフローと処理プロセス

本アプリケーションにおけるデータの流れは、以下の6つのステップで構成されます。

![データフロー図](dataflow.png)

**ステップ1: データ入力** — ユーザーのデバイスから音声、画像、テキスト入力、位置情報、通知データなどがシステムに送信されます。バックグラウンドでの自動収集と、ユーザーの明示的な入力の両方をサポートします。

**ステップ2: 前処理と匿名化** — プライバシー保護のため、個人を特定できる機密情報（第三者の顔、クレジットカード番号等）のマスキングやノイズ除去が、可能な限りエッジデバイス上で行われます。

**ステップ3: AI解析** — クラウド上のAI基盤で、音声のテキスト化（STT）、画像のキャプション生成、自然言語による意味理解・感情分析が実行されます。

**ステップ4: ナレッジグラフ構築** — 解析結果から「誰が」「いつ」「どこで」「何をしたか」「何に興味を持ったか」「どんなスキルを使ったか」という関係性を抽出し、ユーザー個人のナレッジグラフを更新します。

**ステップ5: インサイト生成** — ナレッジグラフを基に、モチベーショングラフの更新、タスクの抽出、プロフィールの再生成、マッチング候補の選定が行われます。

**ステップ6: 出力とフィードバック** — アプリケーションのUIを通じて、プロフィール、グラフ、タスクリスト、おすすめ情報としてユーザーに提示されます。ユーザーからのフィードバック（修正、評価）はモデルの改善に活用されます。

### 9.2 主要画面の詳細

#### ダッシュボード（ホーム画面）

ダッシュボードは、ユーザーがアプリを開いた際に最初に目にする画面です。上部にはユーザーの現在のステータス（気分、活動状況）がアイコンとテキストで表示されます。中央部にはモチベーショングラフの直近1週間のサマリーが折れ線グラフで表示され、タップすると詳細な分析画面に遷移します。下部にはAIが自動抽出した本日のタスクリストが表示され、各タスクはスワイプ操作で完了・延期の操作が可能です。

#### マイプロフィール画面

マイプロフィール画面は、AIが自動生成したユーザーのプロフィールを閲覧・編集する画面です。上部にはプロフィール画像とユーザー名、所属学校が表示されます。その下にはAIが抽出したスキルセットがタグクラウド形式で表示され、各タグをタップすると、そのスキルの根拠となった活動記録を確認できます。経歴セクションでは、活動履歴がタイムライン形式で時系列に表示されます。画面下部には「ポートフォリオとして公開」ボタンと「デジタル名刺を作成」ボタンが配置されます。

#### ディスカバー（マッチング）画面

ディスカバー画面は、AIが提案する「気の合う人」や「おすすめの企業」をカード形式で表示する画面です。上部にはタブ切り替え（友人/恋人/企業/ロールモデル）があり、各タブで異なるレコメンドが表示されます。カードにはレコメンド先のプロフィールサマリーと、マッチング理由（共通の趣味、類似の経歴等）が表示されます。カードをスワイプすることで、「興味あり」「スキップ」の操作が可能です。

## 10. 非機能要件

### 10.1 セキュリティとプライバシー

日常の行動を自動記録するという性質上、極めて高いセキュリティとプライバシー保護が求められます。

| 要件カテゴリ | 要件 | 実装方針 |
| :--- | :--- | :--- |
| **データ暗号化** | 保存データおよび通信経路のすべてを暗号化する | Data at Rest: AES-256, Data in Transit: TLS 1.3 |
| **アクセス制御** | 収集データの種類ごとにユーザーが許可・拒否できる | オプトイン方式、細粒度の権限管理UI |
| **エッジ処理** | 生データのクラウド送信を最小限に抑える | TensorFlow Lite / Core MLによるデバイス上での一次処理 |
| **データ削除権** | ユーザーがいつでも自分のデータを完全に削除できる | GDPR準拠のデータ削除API、30日以内の完全消去 |
| **匿名化** | 第三者の個人情報が記録に含まれる場合の保護 | 顔検出による自動ぼかし、音声の話者分離と匿名化 |
| **監査ログ** | データアクセスの追跡可能性を確保する | すべてのAPIアクセスのログ記録、定期的な監査 |

### 10.2 パフォーマンスとスケーラビリティ

| 要件 | 目標値 | 実装方針 |
| :--- | :--- | :--- |
| **API応答時間** | 95パーセンタイルで200ms以下 | CDN活用、データベースインデックス最適化、キャッシュ戦略 |
| **AI解析処理時間** | 音声1分あたり10秒以下 | 非同期処理、GPUインスタンスの活用 |
| **同時接続数** | 10,000ユーザー以上 | Kubernetesによるオートスケーリング |
| **データ保存容量** | ユーザーあたり月間1GB | S3のライフサイクルポリシーによるコスト最適化 |
| **バッテリー消費** | バックグラウンド記録時、1時間あたり5%以下 | エッジAIの最適化、収集頻度の動的調整 |
| **可用性** | 99.9%（年間ダウンタイム8.76時間以下） | マルチAZ構成、ヘルスチェック、自動フェイルオーバー |

## 11. 運用上の課題と対策

Figmaボードで指摘されている運用上の問題点に対する技術的アプローチは以下の通りです。

### 11.1 データ抽出の精度

音声認識や画像認識の精度がユーザー体験に直結するため、以下の対策を講じます。継続的なモデルのファインチューニングとして、ユーザーのフィードバック（AIが生成した要約やタグの修正）を収集し、モデルの精度向上に活用するフィードバックループを実装します。また、信頼度スコアの活用として、AI解析結果に信頼度スコアを付与し、スコアが低い場合はユーザーに確認を求めるUIを設計します。

### 11.2 プライバシーへの懸念

常時記録という機能の性質上、ユーザーのプライバシーに対する懸念が最大のリスクです。透明性の確保として、どのデータがいつ収集されたかをユーザーがいつでも確認・削除できるダッシュボードを提供します。段階的な権限取得として、初回起動時にすべての権限を一度に求めるのではなく、機能の利用に応じて段階的に権限を要求する設計とします。

### 11.3 運営キャッシュの確保

持続可能なサービス運営のための収益化戦略をシステム設計段階から組み込みます。詳細は次章のビジネスモデルセクションで記述します。
