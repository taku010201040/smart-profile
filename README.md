# Smart Profile 技術仕様書

**バージョン:** 2.0    

---

## 1. プロダクト概要

Smart Profileは、学生の日常活動を自動的にナレッジ化し、AIによる自己分析・プロフィール自動生成を提供するWebアプリケーションである。テキスト・音声・画像による記録入力から、スキル・興味・経験を自動抽出し、モチベーション可視化、マッチング、デジタル名刺生成までを一貫して支援する。本バージョンでは、スマートフォン向けPWA音声レコーダー、ナレッジ共有チャットボット、モチベーショングラフのデモモード・トレンド分析機能を新たに追加した。

### 3.2 テーブル定義

Smart Profileのデータベースは7つのテーブルで構成される。全テーブルはDrizzle ORMのスキーマファイル（`drizzle/schema.ts`）で定義され、`pnpm db:push`コマンドでマイグレーションが実行される。

| テーブル名 | 説明 | 主要カラム |
|-----------|------|-----------|
| `users` | ユーザー情報 | openId, name, email, role, school, grade |
| `activities` | 活動記録 | userId, type(text/audio/image), content, transcription, aiSummary |
| `extracted_insights` | AI抽出インサイト | activityId, userId, type(skill/interest/experience/achievement/goal), label, confidence |
| `profiles` | AIプロフィール | userId, generatedBio, skills(JSON), interests(JSON), strengths(JSON) |
| `tasks` | タスク | userId, title, status, priority, sourceActivityId, isAutoGenerated |
| `connections` | ユーザー接続 | userId1, userId2, status, compatibilityScore |
| `match_recommendations` | マッチング推薦 | userId, targetUserId, type(friend/romantic/career/role_model), score, reasons(JSON) |

### 3.3 データフロー

活動記録の保存からインサイト抽出までのデータフローは以下の通りである。

1. ユーザーがテキスト・音声・画像を入力
2. 音声の場合、Gemini APIで文字起こしを実行
3. 活動記録を`activities`テーブルに保存
4. AI分析エンジンがコンテンツを解析し、スキル・興味・経験を抽出
5. 抽出結果を`extracted_insights`テーブルに保存
6. タスクが検出された場合、`tasks`テーブルに自動登録
7. 蓄積されたインサイトからAIプロフィールを自動生成

### 4.2 External REST API

External REST APIは、PWA音声レコーダーや外部クライアントからのアクセスを想定したRESTful APIである。APIキー認証（Bearer `sp_<openId>`形式）を使用する。

| メソッド | エンドポイント | 説明 |
|---------|--------------|------|
| `POST` | `/api/external/record/text` | テキスト記録作成 |
| `POST` | `/api/external/record/audio` | 音声記録作成（文字起こし付き） |
| `POST` | `/api/external/record/audio/:id/analyze` | 活動のAI分析実行 |
| `GET` | `/api/external/activities` | 活動一覧取得（limit, offset, type対応） |
| `GET` | `/api/external/profile` | プロフィール取得 |
| `GET` | `/api/external/api-key` | APIキー情報取得 |

**認証フロー:**

```
Authorization: Bearer sp_{openId}
```

1. クライアントがAuthorizationヘッダーにAPIキーを付与
2. サーバーが`sp_`プレフィックスを検証
3. openIdでユーザーをデータベースから検索
4. ユーザーが存在すればリクエストコンテキストに`apiUser`を設定

### 5.3 主要コンポーネント

**VoiceRecorder.tsx（820行）** は音声レコーダーのメインコンポーネントであり、以下の機能を実装している。

- **MediaRecorder API**: ブラウザの標準APIを使用してWebM形式で音声をキャプチャする。複数のMIMEタイプ（`audio/webm;codecs=opus`, `audio/webm`, `audio/ogg;codecs=opus`）にフォールバック対応している。
- **自動セグメント分割**: 30秒、1分、2分、5分の4段階から選択可能。セグメント完了時にBase64エンコードしてExternal REST APIに自動送信する。
- **一時停止・再開**: 録音の一時停止と再開に対応し、経過時間の計算を正確に行う。`segmentStartTime`を記録して各セグメントの正確な録音時間を算出する。
- **音声レベル可視化**: Web Audio API（AnalyserNode）を使用してリアルタイムの音声レベルをバー表示する。

**useBackgroundRecording.ts（250行）** はバックグラウンド録音を実現するカスタムフックであり、以下の3つの技術を組み合わせている。

| 技術 | 目的 | 実装 |
|------|------|------|
| Wake Lock API | 画面スリープ防止 | `navigator.wakeLock.request("screen")` |
| Silent Audio Loop | バックグラウンドプロセス維持 | 無音AudioContextの継続再生 |
| Notification API | 録音中の通知表示 | ブラウザ通知で録音状態を明示 |

加えて、`visibilitychange`イベントでアプリのフォアグラウンド/バックグラウンド遷移を検知し、`beforeunload`イベントで録音中のページ離脱を警告する。

### 5.4 文字起こしエンジン

当初はWhisper APIを使用していたが、以下の問題が発生したためGoogle Gemini APIに切り替えた。

| 問題 | Whisper API | Gemini API |
|------|------------|------------|
| ファイルサイズ制限 | 25MB（実質1分程度で制限） | 制限なし（Base64直接送信） |
| ファイル形式 | WebM非対応の場合あり | WebM含む多形式対応 |
| Content-Type | S3経由でのMIME不整合 | Base64直接送信で回避 |
| エラーハンドリング | リトライなし | 3回自動リトライ+指数バックオフ |

**geminiTranscription.ts** の実装では、以下の処理フローを採用している。

1. 音声データをBase64またはURLから取得
2. MIMEタイプを正規化（`normalizeMimeType`関数で12種類のMIMEタイプに対応）
3. Gemini 2.5-flashモデルに`inlineData`として送信
4. 最大3回のリトライ（指数バックオフ: 1秒、2秒、3秒）
5. 文字起こし結果をテキストとして返却

### 6.3 フロントエンド実装

**KnowledgeChatbot.tsx** はフローティングチャットボットUIを提供するコンポーネントである。

- **フローティングアイコン**: 画面右下に固定配置（`fixed bottom-4 right-4 z-50`）。未読メッセージがある場合はパルスアニメーション付きの赤いバッジを表示する。
- **チャットウィンドウ**: 380px幅、520px高さのカード型ウィンドウ。最小化・閉じるボタンを備える。
- **AIChatBoxコンポーネント**: メッセージ履歴の表示、Markdown対応のレンダリング（Streamdownライブラリ）、サジェストプロンプト（4種類）を提供する。
- **CustomEventリスナー**: `open-chatbot`イベントを監視し、モチベーショングラフの「深掘り」ボタンからの質問を受け取る。

**App.tsx** では`ChatbotWrapper`コンポーネントを定義し、認証済みユーザーにのみチャットボットを表示する。`useAuth()`フックでユーザー状態を確認し、未認証時は`null`を返す。

### 6.4 バックエンド実装

`chatbot.chat`プロシージャは以下の処理を行う。

1. **データ収集**: 4つのデータソースを並列取得（`Promise.all`）
   - 直近30件の活動記録
   - 全インサイト
   - プロフィール情報
   - 直近20件のタスク

2. **コンテキスト構築**: 収集したデータをMarkdown形式のテキストに変換し、システムプロンプトに埋め込む。

3. **LLM呼び出し**: Manus LLM APIに対して、システムプロンプト（ナレッジコンテキスト含む）とユーザーメッセージ履歴を送信する。

4. **レスポンス返却**: LLMの回答をフロントエンドに返却する。

### 6.5 モチベーショングラフとの連携

Motivation.tsxのトレンドサマリーやツールチップから、`CustomEvent`を通じてチャットボットに質問を投げることができる。

```typescript
// Motivation.tsx
const handleAskChatbot = useCallback((question: string) => {
  window.dispatchEvent(new CustomEvent("open-chatbot", { detail: { question } }));
}, []);
```

この連携により、グラフ上のトレンド変動（上昇・下降）の理由をチャットボットに深掘りできる。

## 7. モチベーショングラフ

### 7.1 概要

モチベーショングラフは、ユーザーの活動量を時系列で可視化し、トレンド分析を提供する機能である。Rechartsライブラリを使用して、棒グラフ（日別活動量）、エリアチャート（累積トレンド）、円グラフ（インサイト分布・タスク達成状況）を描画する。

### 7.2 デモモード

データが不足している場合にサンプルデータを表示するデモモード機能を実装した。

- **トグルスイッチ**: 画面右上にデモモード切り替えスイッチを配置。オン時は「サンプルデータ表示中」バッジを表示する。
- **仮想データ生成**: `generateDemoData()`関数が30日分の活動データを生成する。曜日による変動（週末は40%に減少）、8つのイベント（プロジェクト開始、体調不良、連休等）による変動を含む。
- **安定参照**: `useState(() => generateDemoData())`パターンで初期化し、レンダリングごとの再生成を防止する。

### 7.3 トレンド検出

`detectTrends()`関数が日別データの変曲点を検出する。前日比で40%以上の変動がある場合にトレンド（上昇/下降）としてマークし、理由テキストを付与する。

### 7.4 カスタムツールチップ

`CustomBarTooltip`と`CustomAreaTooltip`の2つのカスタムツールチップコンポーネントを実装した。

- テキスト・音声・画像の内訳表示
- 合計値の表示
- トレンド理由の表示（上昇は緑、下降は赤のアイコン付き）
- 「チャットボットで深掘り」リンク（クリックでチャットボットに質問を送信）

## 8. 認証・セキュリティ

### 8.1 Manus OAuth

メインアプリケーションの認証にはManus OAuthを使用する。`/api/oauth/callback`でOAuthフローを完了し、セッションCookieを発行する。`protectedProcedure`でラップされたtRPCプロシージャは、自動的に認証チェックを行う。

### 8.2 APIキー認証

External REST APIでは、`Bearer sp_<openId>`形式のAPIキー認証を使用する。APIキーはユーザーのopenIdから一意に生成され、設定画面（`/settings`）で確認・コピーできる。

### 8.3 入力バリデーション

全てのtRPCプロシージャ入力はZodスキーマで検証される。External REST APIでも同様に、リクエストボディの検証を行う。

## 11. 既知の制限事項と今後の課題

### 11.1 既知の制限事項

| 項目 | 説明 |
|------|------|
| バックグラウンド録音 | iOSのSafariではWake Lock APIが制限されるため、バックグラウンド録音が中断される可能性がある |
| Gemini APIレートリミット | 無料枠では1日20リクエストの制限があり、頻繁な文字起こしには有料プランが必要 |
| チャットボットのコンテキスト長 | 活動記録が多い場合、LLMのコンテキストウィンドウを超える可能性がある |
| デモモードのデータ | 仮想データは固定パターンであり、実データとの乖離がある |

### 11.2 今後の課題

- **ストリーミングレスポンス**: チャットボットの回答をストリーミングで表示し、体感速度を向上させる
- **オフライン対応強化**: 録音データのローカルキューイングとオンライン復帰時の自動同期
- **プッシュ通知**: Web Push APIを使用したタスクリマインダー・マッチング通知
- **マルチモーダル分析**: 画像記録のAI分析（OCR、シーン認識）の強化
- **エクスポート機能**: プロフィール・活動記録のPDF/CSV出力
